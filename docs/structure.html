第2章 名前空間の設計

2.1 名前空間はブランドである

名前空間は「フォルダ名」ではなく「あなたの署名」。Smithed規約は ファイルパス・タグ・スコア・チーム・storageすべてに一貫したプレフィックスを要求している。 パック名/作者名/略称のいずれか、一貫して自分に紐づくものを選ぶ。

tcc          # "The Creeper's Code" の略(Smithedの実例)
rx.playerdb  # 作者グループ rx-modules + パック名(実例)
gm4_xxx      # Gamemode 4 のモジュール群
bs           # Bookshelf

推奨パターン:

  • 単発パック: niwatori_casino のようなパック名そのまま
  • 同一作者の複数パック群: nw.casino / nw.pvp のように 作者略称.モジュール で階層化
  • scoreboard等の16文字制限があるものは2〜4文字の短縮形(nw)、 ファイルパスやタグはフル(niwatori)と、長短2つの正式名を最初に決めて表に書いておく

2.2 minecraft名前空間は「上書き専用」

data/minecraft/ に置いてよいのは:

  • tags/function/load.json, tags/function/tick.json への 追記
  • バニラのloot table / recipe / tag等を意図的に置き換える場合

やってはいけないこと(MC Datapacks規約が明示的に禁止):

  • tick.json / load.json に "replace": true を設定する
    → 共存する他のデータパック全てのtickを殺す。サーバー運営では致命傷。
  • 自作関数を data/minecraft/function/ に置く(衝突・上書き事故のもと)
// ✅ data/minecraft/tags/function/tick.json
{
  "values": [
    "niwatori:core/tick"
  ]
}

2.3 1データパック=1名前空間が基本、例外は「意図的な公開I/F」

基本は自分の名前空間1つ。例外として、他パックとの連携用グローバル名前空間 (global: や #smithed.xxx のような規約タグ)に参加する場合のみ複数名前空間を持つ。 「なんとなく機能ごとに名前空間を分ける」のは、内部向けならフォルダ分けで十分。 名前空間の数はインターフェースの数、フォルダはモジュールの数、と考える。

第3章 ディレクトリ構造と関数の分割

3.1 レイヤーを分ける: エントリポイント / API / 内部実装

読める大規模データパックに共通するのは「どこから実行が始まり、どこが外部公開で、 どこが触ってはいけない内部か」が構造で分かること。IMP-Docのアクセス修飾子 (@public / @api / @within / @private)の考え方をフォルダ構造に落とすとこうなる:

data/niwatori/function/
├── load.mcfunction            # エントリポイント: #minecraft:load から
├── tick.mcfunction            # エントリポイント: #minecraft:tick から
├── api/                       # 他パック・コマブロから呼んでよい公開API
│   ├── give_money.mcfunction
│   └── open_atm.mcfunction
├── casino/                    # モジュール(機能単位)
│   ├── atm/
│   │   ├── open.mcfunction
│   │   ├── deposit.mcfunction
│   │   └── _render_balance.mcfunction    # 先頭 _ = このフォルダ内部専用
│   └── slot/
├── pvp/
│   ├── rating/
│   └── arena/
├── core/                      # 全モジュール共通の基盤(tick分配、時計、定数)
│   ├── tick.mcfunction
│   └── constants.mcfunction
└── debug/                     # デバッグ用(配布時に消せる独立性を保つ)

ポイント:

  • tick.jsonに直接ぶら下げる関数は1つだけにして(datapack.wikiも推奨)、 その中で各モジュールのtickへ分配する。tickの全体像が1ファイルで読める。
  • 内部専用関数は _ プレフィックスや impl/ zz_ フォルダで「呼ぶな」を表現する 流儀がある(mcfunctionに言語レベルのprivateはないため、命名で代用する)。
  • debug/ test/ は本体から参照されない形で分離し、リリース時に丸ごと消せるようにする。

3.2 1ファイル=1責務、「executeの都合」による分割を恐れない

mcfunctionでは以下の理由でファイル分割が他言語より頻繁に必要になる:

  1. execute as @e[...] run function ... で実行者を切り替える(@s化)
  2. 条件分岐のブロック化(if文の中身に相当する部分を別ファイルへ)
  3. マクロ行を含む部分の分離(第6章)
  4. 再帰ループの本体

これは「悪」ではなく、mcfunctionにおける関数抽出(Extract Method)そのもの。 むしろ「このファイルは何の文脈で実行されるか」が一意になるように割るのが正しい。 1ファイル内に「サーバーとして実行される行」と「各プレイヤーとして実行される行」が 混ざっている方がよほど読めない。

❌ tick.mcfunction 内で
execute as @a[predicate=nw:is_in_casino] at @s run tp ...
execute as @a[predicate=nw:is_in_casino] at @s run particle ...
execute as @a[predicate=nw:is_in_casino] at @s run playsound ...

✅ tick.mcfunction:
execute as @a[predicate=nw:is_in_casino] at @s run function nw:casino/player_tick
   casino/player_tick.mcfunction:  (@s = カジノ内プレイヤー, at @s)
tp ...
particle ...
playsound ...

セレクタ評価が1回になるので速度も上がる(第9章)。可読性と性能が同じ方向を向く珍しい言語。

3.3 フォルダ名は複数形/カテゴリ名、ファイル名は単数の動作

casino/games/slot/spin.mcfunction
casino/games/poker/deal.mcfunction

vanillaデータ(loot_table等)の構造も参考になる。chests/, entities/ のように バニラはカテゴリフォルダを複数形にしている。

3.4 function以外のリソースも同じ木構造にそろえる

predicate / advancement / loot_table を、対応するfunctionと同じサブパスに置くと 関連が追いやすい:

data/niwatori/function/casino/atm/open.mcfunction
data/niwatori/predicate/casino/is_in_casino.json
data/niwatori/advancement/casino/atm/interacted.json
data/niwatori/loot_table/casino/reward/jackpot.json

進捗トリガー→rewards.function→関数、の連鎖はデータパックの主要イベント機構なので、 advancement/casino/atm/interacted.json と function/casino/atm/on_interact.mcfunction のように 対になる名前を付ける(イベントハンドラ命名: on_<イベント>)。

3.5 生成ファイルは手書きファイルと分ける

Pythonスクリプト等でloot tableやmcfunctionを自動生成する場合(チェスト戦利品テーブル等)、 生成物は generated/ サブフォルダに隔離し、フォルダ先頭に # THIS FILE IS GENERATED BY tools/chest-loottable.py -- DO NOT EDIT を入れる。 Bookshelfもビルド時生成ファイルを明確に区別している。生成物を手で直すと次の生成で消えるので、 「手で編集してよい領域」と「スクリプトの出力先」を物理的に分離するのが鉄則。
