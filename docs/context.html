第4章 コメントとドキュメンテーション

4.1 全関数にヘッダコメントを書く(GM4標準)

Gamemode 4はテンプレート由来以外の全関数に以下のヘッダを義務付けている。 mcfunctionには引数リストも型もdocstringも無いので、これが唯一の「シグネチャ」になる:

# Zauber大釜の生成時にパーティクル演出を再生する
# @s = 付呪本を持っているプレイヤー
# at @s (align xyz)
# run from niwatori:cauldron/create

最低限書くべき4点:

  1. 何をする関数か(1行説明)
  2. @s が誰か(セレクタ表記 or 日本語。複雑なセレクタは日本語で)
  3. 実行位置/回転(at @s, positioned 0 64 0, alignの有無など)
  4. どこから呼ばれるか(run from nw:xxx、複数なら「functions in nw:till/blocks/」、 scheduleなら「scheduled by nw:xxx」)

「どこから呼ばれるか」はgrepでも調べられるが、ヘッダにあることで 開いた瞬間に文脈が復元できる。これがmcfunction可読性の生命線。

4.2 IMP-Doc: 機械可読なドキュメント規約

日本コミュニティ発で広く使われるドキュメント仕様 IMP-Doc(Interoperable Module Protocol Documentation)は、上のヘッダコメントを形式化したもの。先頭行を #> で始める:

#> niwatori:casino/atm/deposit
# ATMへの入金処理。所持金スコアをstorageの口座残高へ移す。
#
# @within function niwatori:casino/atm/menu_handler
# @input
#   score @s nw.deposit_amount : 入金額(タマ)
# @output
#   storage niwatori:data account[].balance : 更新後残高
# @private

主なアノテーション(IMP-Doc仕様 + DHP拡張):

アノテーション意味
@input / @output実行前に設定すべき値 / 実行後に設定される値
@within <パス>指定した場所からしか呼ばれない(実質private)
@private / @public / @apiアクセス修飾子
@userユーザー(プレイヤーやコマブロ)が直接実行してよい
@context要求する実行文脈(位置・回転・実行者)
@handles / @patchイベント処理 / 他リソースへのパッチ
@deprecated / @internal非推奨 / 内部用(DHP拡張)

エディタ(Spyglass / Datapack Helper Plus)がこれを解釈して補完・参照検索に使える。 公式仕様ではなくただのコメントなので、ゲーム動作には一切影響しない=導入コストゼロ。

4.3 処理ブロックごとの節コメント

Elliversのプラクティス集が示す標準形。空行+コメントで段落を作る:

#> niwatori:pvp/arena/start
# 試合開始処理
# @within function niwatori:pvp/lobby/countdown

# 参加者を初期化
scoreboard players set @a[tag=nw.pvp.player] nw.pvp.kills 0
effect clear @a[tag=nw.pvp.player]

# アリーナへ転送
execute as @a[tag=nw.pvp.player] run function niwatori:pvp/arena/teleport_to_spawn

# 開始演出
title @a[tag=nw.pvp.player] title {"text":"FIGHT","color":"red"}

4.4 コメントの内容: 「何を」より「なぜ」

コマンドの直訳コメント(# プレイヤーにダイヤを与える → give @s diamond)は書かない。 書くべきはコマンドから読み取れない理由・制約・ハック:

# NG: tpでY+0.5している           ← 見れば分かる
# OK: マーカーが半ブロック床にめり込むとレイキャストが失敗するためY+0.5補正
tp @s ~ ~0.5 ~

特にmcfunctionは「1tick遅延しないと反映されないNBT」「セレクタの評価順」「マクロの再コンパイル」 などの罠ハック密度が高いので、未来の自分への言い訳を必ず残す。

4.5 #region による折りたたみ

長い分岐列挙は #region / #endregion(DHP/一部エディタが折りたたみ対応)でグループ化できる:

#region スロット絵柄ごとの配当分岐
execute if score #reel nw.tmp matches 1 run function nw:casino/slot/payout/cherry
execute if score #reel nw.tmp matches 2 run function nw:casino/slot/payout/bell
execute if score #reel nw.tmp matches 3 run function nw:casino/slot/payout/seven
#endregion

ただし折りたたみが必要なほど長い列挙は、そもそも分岐テーブル関数への分割やマクロ化を検討するサイン。

4.6 コメントも保守対象

Bookshelfのガイドラインが強調する点: 挙動を変えたらコメントも更新する。 古いコメントは無いより悪い。特に @input/@output とヘッダの「run from」は リファクタ時に嘘になりやすいので、レビュー/セルフチェック項目に入れる(第16章)。

第5章 実行文脈(execution context)の設計

5.1 「@s中心主義」で書く

mcfunctionの可読性を最も左右するのは実行文脈の管理。原則:

  1. 関数の中では@sと相対座標だけを使う(誰に対して動くかは呼び出し側が決める)
  2. セレクタで対象を選ぶのは呼び出し側(分配層)の責務
  3. 文脈を変えるexecuteが3行以上続いたら関数に切り出す

これにより関数が「対象非依存の部品」になり、テストも再利用も楽になる。 Bookshelfの「Conservation Principle」も同じ思想で、関数は入力に責任を持ち、 出力以外を書き換えない(勝手にスコアを消したり作ったりしない)ことを求めている。

5.2 文脈をヘッダに書く(第4章と連動)

# @s = ... # at @s が書いてあれば、~ ~ ~ が誰の足元かが一目で分かる。 文脈が「サーバー(実行者なし)」の関数では、うっかり@sや相対座標を使わないこと。 loadやscheduleから呼ばれる関数は実行者・位置を失う(ワールドスポーン基準になる)ことを ヘッダに明記しておくと事故が減る:

#> niwatori:core/minute_clock
# scheduleで毎分self-rescheduleされる時計
# ⚠ 実行文脈なし: @sは存在せず、位置はワールドスポーン。エンティティ操作禁止

5.3 anchoredとalignは使った場所でコメント

anchored eyes や align xyz は後続コマンドの座標の意味を静かに変える。 使う場合は同一行 or 直前コメントで意図を書く。
