第7章 変数管理(scoreboard / storage / マクロ引数)

7.1 使い分けの原則

手段型速度用途
scoreboardint32のみ最速カウンタ、フラグ、演算、tick毎の状態
storage任意NBT速い(entity/block NBTより大幅に)構造化データ、文字列、リスト、設定
entity/block NBT任意遅い(特にプレイヤー)最終読み書きのみ。作業領域にしない
マクロ引数任意呼び出しが遅い関数への引数渡し専用

NBTを何度も加工するときは「一度storageへコピー→storage上で加工→書き戻し」が定石 (Elliversのプラクティス集)。

7.2 objectiveは増やしすぎない: フェイクプレイヤー活用

「変数1個ごとにobjectiveを1個作る」のは典型的アンチパターン。 グローバル変数はobjective 1〜2個(nw.var 等)+フェイクプレイヤー名で無限に作れる:

scoreboard players set #casino.jackpot nw.var 0
scoreboard players set #arena.state nw.var 1

objectiveを分けるべきなのは「エンティティごとに値を持つ」「trigger用」「statistics連動」の時。

7.3 定数はloadで初期化する定数プール

乗除算に使う定数は nw.const objectiveに #値 名で集中管理(1.5参照)。 loadで必ず初期化し、実行中にsetし直さない:

#> niwatori:core/constants
# @within function niwatori:load
scoreboard players set #2 nw.const 2
scoreboard players set #100 nw.const 100
scoreboard players set #20 nw.const 20    # 1秒 = 20tick

7.4 「0と未定義は違う」を設計に織り込む

matches ..0 は未定義スコアにマッチしない。テストワールドでは動くのに本番で動かない バグの代表格。プレイヤー系スコアは参加時/loadで明示的に0初期化する規約にする:

# 新規参加者の初期化(tickで未初期化者だけ拾う)
execute as @a[predicate=!nw:is_initialized] run function nw:core/init_player

7.5 一時変数の規約

  • 名前: #tmp, #tmp2 ではなく #calc.damage, #calc.rate のように用途を書く。 ただし関数内で完結し、次のtickに持ち越さないものだけを一時変数と呼ぶ。
  • 使い終わった一時storageは放置でよい(次回上書きされる設計にする)。 逆に data remove で几帳面に消して回るとコスト増+他所が読んでいた場合に事故る。 Bookshelfの保存原則: スコアの削除は極力しない、リセット/上書きで済ます。

7.6 enumは数値+コメント表 or 「Enum Scoreboard」

状態機械(ゲーム進行state等)は文字列でなくintで持ち、対応表をコメントとloadに残す:

#> niwatori:pvp/arena/states
# #arena.state nw.var の値:
#   0 = 待機 / 1 = カウントダウン / 2 = 試合中 / 3 = リザルト

MC Datapacksコミュニティには、値名をフェイクプレイヤー化してobjective自体を enum定義として使う「Enum Scoreboard」というテクニックもある (scoreboard players set WAITING nw.arena_state_enum 0 のように定義を実体化する)。

第8章 制御構造のイディオム

8.1 早期リターン(return)を使う

1.20.2+の /return はガード節そのもの。ネストしたexecuteより読める:

#> niwatori:casino/atm/withdraw
# 出金。残高不足なら即終了
execute if score @s nw.money < #withdraw.amount nw.var run return run \
    tellraw @s {"text":"残高が足りません","color":"red"}

# ここから正常系
...

Minecraft Wiki最適化チュートリアルも「後続条件があり得なくなったらreturn runで打ち切る、 順序に意味がなければ成立確率の高い条件を先に書く」ことを推奨。

⚠ 注意: execute as @e ... run return は最初の1体で関数全体が終わるので、 複数対象への分岐がある関数では使わない(分岐専用関数に隔離してから使う)。

8.2 if/else は「フラグ+2連if」か「return」で

mcfunctionにelseは無い。素朴な2連 if/unless は1行目が状態を変えると2行目も走る事故がある。 定石は (a) 結果を一時スコアに保存して分岐、(b) return方式:

# (a) 判定を先に固定する
execute store success score #can_pay nw.tmp if score @s nw.money matches 100..
execute if score #can_pay nw.tmp matches 1 run function nw:casino/slot/spin
execute if score #can_pay nw.tmp matches 0 run tellraw @s {"text":"コインが足りない"}

8.3 ループは3種類を使い分け、必ず停止条件を先頭に

種類実装用途
毎tickループ#minecraft:tick常時監視。最小限に
低頻度ループschedule function ... 20t の自己再スケジュール秒単位で十分な監視・時計
tick内再帰関数が自分をfunctionで呼ぶカウント処理、レイキャスト

再帰は必ず1行目に脱出条件(ガード)を書く。条件なし再帰はmaxCommandChainLength まで暴走する。レイキャストの1ステップ距離は貫通防止のため0〜1ブロック程度にする (コミュニティの定番注意点)。scheduleは実行文脈を失う点もヘッダに明記(5.2)。

8.4 分岐テーブルは「ディスパッチ関数」に隔離

同じ対象への execute if score ... run function ... の列挙は、それ専用の dispatch.mcfunction / payout/_dispatch.mcfunction を作ってそこに置く。 tick関数や業務ロジックの中に分岐列挙を埋め込まない。

8.5 イベント駆動を最優先する(ポーリングは最後の手段)

「毎tick全員を調べる」前に、advancementトリガー(player_interacted_with_entity, placed_block, inventory_changed, using_item...)+ rewards.function で 起きた時だけ関数を走らせる設計を検討する(Elliversも同旨)。 進捗は revoke して使い捨てるイディオムとセットで:

#> niwatori:casino/atm/on_interact
# advancement niwatori:casino/atm/interacted の rewards から
# @s = ATMエンティティをクリックしたプレイヤー
advancement revoke @s only niwatori:casino/atm/interacted
function niwatori:casino/atm/open

triggerスコアボード(dialogのrun_command権限回避等)も同様にイベント駆動の一種。 tick監視 → scoreboard/predicate → advancement/trigger の順で「より受動的な仕組み」へ 置き換えられないか常に考える。
