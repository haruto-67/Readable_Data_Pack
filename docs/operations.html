第11章 load / tick / インストール / アンインストール

11.1 標準ファイルセットを決める

良いデータパックは「入口」が全部同じ場所にある。推奨テンプレート:

function/
├── load.mcfunction        # #minecraft:load から。冪等に書く
├── tick.mcfunction        # #minecraft:tick から。分配のみ
├── install.mcfunction     # 初回セットアップ(forceload, marker設置等)
├── uninstall.mcfunction   # 完全撤去
└── version.mcfunction     # バージョン表示(任意)

Bookshelfは __load__ / __unload__ / __help__ をモジュール毎の予約名にしている。 名前は何でもよいが全モジュールで統一する。

11.2 loadは冪等(何度実行しても安全)に

/reload のたびに走るので、loadで「追加」する処理は既存チェック付きにする:

#> niwatori:load
# ⚠ /reloadごとに実行される。冪等であること
scoreboard objectives add nw.var dummy
scoreboard objectives add nw.money dummy "所持金"
function niwatori:core/constants
schedule function niwatori:core/minute_clock 1t replace   # replaceで多重登録防止
tellraw @a[tag=nw.admin] {"text":"[にわとりMC] reloaded vX.Y.Z","color":"gray"}

schedule ... replace を使わないとreloadのたびにループが増殖する。定番バグ。 reload完了メッセージ(バージョン付き)は運用上のデバッグ効率を大きく上げる (MC Datapacksスタイルガイドの「Reloading Message」)。

11.3 uninstallを提供する

MC Datapacks規約「Datapack Uninstallation」。行儀の良いパックは 自分が作ったものを全部消す関数を持つ:

#> niwatori:uninstall
# このデータパックの痕跡を全て除去する。実行後にdatapack disable推奨
scoreboard objectives remove nw.var
scoreboard objectives remove nw.money
kill @e[tag=niwatori.entity]
data remove storage niwatori:data {}
schedule clear niwatori:core/minute_clock
tellraw @s {"text":"[にわとりMC] アンインストール完了。datapack disableしてください"}

サーバー運営では「入れて試して消す」が頻繁なので、uninstallの有無が事故率に直結する。

第12章 バージョン管理・配布・メタ情報

12.1 pack.mcmeta

  • description にはパック名・対象MCバージョン・自分のバージョン・作者を入れる (データパック選択画面とdatapack listで見える唯一の自己紹介)。
  • 1.21.9以降(pack_format 88〜)は min_format / max_format 方式に移行済み。 1.21.11なら新旧両方の書式に注意。26.x系(pack_format 100+)への更新時は フォーマット変更点(NBT型厳格化等)の洗い出しをリリースノートで確認する。
{
  "pack": {
    "description": "にわとりMC Casino v1.4.0 (MC 1.21.11) by niwatori",
    "min_format": 88,
    "max_format": 88
  }
}

12.2 バージョン番号をデータでも持つ

load時に #version.major nw.var 等へ格納しておくと、 ワールドに残った旧データからのマイグレーション(「v1.3のstorage形式をv1.4へ変換」)が書ける。 GM4がモジュール更新リストで同様の仕組みを持つ。storage構造を変えたら移行関数を書く、 を運用ルールにする。

12.3 Gitで管理する

  • リポジトリのルートは「pack.mcmetaがある階層」(エディタ支援ツールの前提でもある)。
  • zipではなくフォルダのままworldに置き、シンボリックリンク or デプロイスクリプトで反映。
  • コミット規約(Bookshelfは Conventional Commits 風)を軽く決めておくとchangelogが自動化できる。

第13章 開発環境・ツール・CI

  • VSCode + Spyglass(旧Datapack Helper Plus): 補完・参照検索・リネーム・IMP-Doc解釈。 データパック開発の事実上の標準環境。
  • language-mcfunction(Arcensoth): シンタックスハイライト。
  • datapack-linter(ChenCMD): GitHub Actionsでpush毎に全mcfunction/jsonの構文チェック。 ゲームを起動せずに壊れたコマンドを検出できる。マルチ人数開発やリファクタ時の保険に最適。
  • misode.github.io: loot table / predicate / advancement等のGUIジェネレータ。
  • ビルドツール: beet(GM4が採用)/ sandstone / mcbuild など。テンプレ大量生成や 複数パックの合成が必要な規模になったら導入を検討。Pythonでの自前生成(3.5)も立派な選択肢。
  • Weld(Smithed): 複数データパック/リソパの合成ツール。配布時の互換性検証にも。
