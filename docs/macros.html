第6章 マクロ関数の設計 ★

本ガイドの目玉。マクロは強力だが「毎回再解析される」「ファイルが散らかる」という代償がある。設計規約で制御する。

1.20.2で追加されたマクロ($(name) 展開)は強力だが、 「マクロ行を含む関数は呼び出し時に毎回再解析される」という性質があり、 datapack.wikiやコミュニティの共通見解として通常のコマンドより大幅に遅い。 さらに「マクロを使う行は必ず専用ファイルに分けたくなる」ため、設計規約が無いと ファイルが散らかる。ここが本ガイドの目玉。

  • 6.1 「本体」と「マクロ実体」をペアにする

6.1 命名規約: 「本体」と「マクロ実体」をペアにする

マクロのために分離したファイルは、呼び出し元と同名+固定サフィックス/固定サブフォルダにする。 おすすめは2案。プロジェクトでどちらかに統一する:

案A: 同名フォルダ + macro.mcfunction(推奨)

casino/atm/deposit.mcfunction          # 引数を準備して呼ぶ「本体」
casino/atm/deposit/macro.mcfunction    # $(...)を含む「マクロ実体」

案B: サフィックス方式

casino/atm/deposit.mcfunction
casino/atm/deposit_macro.mcfunction    # または deposit.m.mcfunction

どちらでも「deposit を読みたい人が迷わず両方を見つけられる」ことが目的。 逆に マクロ実体を util/ や macro/ などの遠いフォルダにまとめて置かないこと。 処理のまとまり(モジュール)が壊れ、コードジャンプが往復になる。

さらにIMP-Docの @within を書けば「このmacroは隣のdepositからしか呼ばれない」ことが 機械にも人間にも伝わる:

#> niwatori:casino/atm/deposit/macro
# 実際の入金額を反映するマクロ実体
# @within function niwatori:casino/atm/deposit
# @input
#   args: {amount: int, unit: string}
$scoreboard players remove @s nw.money $(amount)
$tellraw @s {"text":"$(amount) $(unit) を入金しました"}

6.2 引数受け渡しの呼び出し規約を1つに固定する

マクロの引数は「インラインNBT」「with storage」「with entity」「with block」の4通りあるが、 プロジェクト内の規約としては次の2択に絞ると読みやすい:

  1. 定数引数 → インラインNBTで直書き(呼び出し箇所を見れば値が分かる) 
    function nw:casino/atm/deposit/macro {amount: 100, unit: "タマ"}
  2. 動的引数 → 専用storage nw:in の関数パスと同じキーに詰めて with で渡す 
    # 呼び出し側(本体 deposit.mcfunction)
execute store result storage nw:in casino.atm.deposit.amount int 1 \
    run scoreboard players get @s nw.deposit_request
data modify storage nw:in casino.atm.deposit.unit set value "タマ"
function nw:casino/atm/deposit/macro with storage nw:in casino.atm.deposit

「引数はどこから来たのか」問題(マクロ最大の可読性リスク)を、 nw:in の casino.atm.deposit を見れば必ず分かるという形で解決する。 Bookshelf等のライブラリも「入力→専用の場所に置く→呼ぶ」の規約で統一している。 戻り値が要る場合は同様に nw:out へ書く or return run で数値を返す。

6.3 マクロ行は最小限に隔離する

再解析コストはマクロ行を含むファイル全体にかかる。したがって:

❌ 40行の処理の中に $行が2行混ざったファイル
✅ 38行の通常関数 + $行2行だけのマクロ実体ファイル

「$が1行でもあるならそのファイルは短く」を徹底する。 また、マクロで実現できても、分岐数が少なく固定なら普通のexecute分岐の方が速い。

❌ 3種類しかない絵柄をマクロで動的ディスパッチ
✅ execute if score ... matches 1 run function nw:...cherry (3行書く)

マクロが正当化されるのは「値が実行時にしか決まらない」「組み合わせが多すぎて列挙不能」 「tellraw/give等の文字列に動的値を埋める」場合。 tick毎に走るホットパスからのマクロ呼び出しは設計から見直す。

6.4 マクロ引数名の命名

  • $(a) $(x) のような1文字は禁止。呼び出し側のNBTキーがそのままドキュメントになるので $(amount) $(target_uuid) $(dimension) のようにキー名=引数名として意味を持たせる。
  • 単位を含める: $(delay_ticks), $(price_tama)。
  • 同じマクロ実体を複数箇所から呼ぶ場合、引数構造(キー名と型)はヘッダの @input に必ず書く。 マクロは引数不足だと実行時に初めて失敗する(コンパイル時チェックがない)ため、 ドキュメントが型定義の代わりになる。

6.5 セキュリティ: ユーザー入力をマクロに流さない

$$(code) のような書き方や、プレイヤーが編集可能なNBT(記名アイテム、本の内容、 custom_data)を無検証でマクロに渡すと、任意コマンド実行(ACE)が成立し得る。 コミュニティでも実証されている既知の穴。サーバー運営では特に:

  • マクロに渡す文字列は自分のデータパックが書き込んだものだけに限定する
  • プレイヤー由来の文字列を使うなら、事前にホワイトリスト照合(スコアIDやenum化)を挟む
  • 「文字列そのもの」ではなく「ID番号→loadで初期化したテーブル参照」で間接化する

6.6 マクロvs代替手段の選択表

やりたいこと第一候補マクロを使うべき場合
少数の固定分岐execute if + 分岐関数分岐が数十以上で列挙が破綻する時
動的な数値をコマンドへexecute store / scoreboardtp座標やeffect時間など store不可の引数
動的テキスト表示tellrawのscore/nbt/selector埋め込み文体自体が変わる・翻訳分岐など
アイテム動的生成loot table + item_modifierNBT構造が実行時にしか決まらない時
エンティティ指定tag付け→セレクタUUIDを直接コマンドに埋める必要がある時

「マクロでしかできないこと」以外にマクロを使わない、が結論。
