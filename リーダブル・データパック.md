# リーダブル・データパック
## 〜良いデータパックの書き方 完全ガイド〜

対象バージョン: Java Edition 1.21.x（1.20.2以降のマクロ対応を前提。26.x系への移行時の注意も併記）

このドキュメントは『リーダブルコード』のようなコーディング作法本の「Minecraftデータパック版」として、
コミュニティの規約（Smithed Conventions / MC Datapacks Conventions / Gamemode 4 標準 /
Bookshelf コントリビューションガイド / IMP-Doc / Minecraft Wiki 最適化チュートリアル 等）と、
GitHub上の著名データパック（Gamemode 4, Bookshelf, PlayerDB, Vanilla Tweaks系 等）の実践から
「こう書かれたデータパックは読みやすく、壊れにくく、速い」というルールをまとめたものです。

> **一番大事な原則（リーダブルコードと同じ）**
> コードは「書く時間」より「読む時間」の方が圧倒的に長い。
> mcfunctionは特に *実行文脈（誰が・どこで実行しているか）がファイルに書かれていない* 言語なので、
> 「未来の自分が3秒で文脈を復元できるか」を常に基準にする。

# 1. 命名規則（ネーミング）

## 1.1 基本は lower_snake_case

ファイル名・フォルダ名・名前空間・リソースパスは **小文字スネークケース**（`lower_case_with_underscores`）。
これはMinecraft Wiki公式チュートリアルでも推奨されており、そもそもリソースロケーションに使える文字が
`0-9 a-z _ - . /` に制限されているため、大文字は物理的に使えない。

```
✅ data/niwatori/function/casino/atm/open_menu.mcfunction
❌ data/Niwatori/function/Casino/ATM/OpenMenu.mcfunction   # そもそも読み込まれない
❌ data/niwatori/function/casino/atm/openmenu.mcfunction   # 単語の切れ目が読めない
```

ハイフン `-` も使えるが、コミュニティの主流はアンダースコア。**混ぜないこと**が最重要。
プロジェクト内で `open-menu` と `open_menu` が混在すると、補完に頼れなくなりtypoバグの温床になる。

## 1.2 「何をするか」が読める動詞始まりの関数名

関数は「実行するもの」なので、**動詞_目的語** が基本形。

```
✅ give_reward.mcfunction        # 報酬を与える
✅ check_balance.mcfunction      # 残高を確認する
✅ reset_arena.mcfunction        # アリーナをリセットする
❌ reward.mcfunction             # 与えるのか、定義なのか、判定なのか不明
❌ arena2.mcfunction             # 2とは
```

ただし「フォルダ名が目的語、ファイル名が動詞」という分担も広く使われる。
むしろ規模が大きくなるとこちらの方が読みやすい:

```
casino/atm/open.mcfunction       # atmを開く（casino:atm/open）
casino/atm/close.mcfunction
casino/atm/deposit.mcfunction
casino/atm/withdraw.mcfunction
```

リソースパス全体（`casino:atm/open`）を1つの文として読んだとき意味が通るように名付ける。

## 1.3 状態を返す関数・判定系は疑問形/状態名

```
✅ is_sneaking(predicate)
✅ has_ticket(predicate)
✅ can_afford.mcfunction     # 残高判定してスコアに結果を返す関数
```

predicateはまさに真偽値なので `is_` `has_` `can_` プレフィックスが読みやすい。

## 1.4 スコアボードobjectiveの命名

- **objective名は16文字制限**がある（表示名は別）。この制約下で名前空間衝突を避ける必要がある。
- Smithed規約・GM4標準・Bookshelf規約はいずれも「**短縮プレフィックス + `.` or `_` + 名前**」を要求する。

```
✅ nw.money          # nw = niwatori の短縮名前空間
✅ nw.pvp_rating
✅ gm4_speed         # Gamemode 4 は gm4_ プレフィックス
✅ bs.var            # Bookshelf は bs. プレフィックス
❌ money             # 他のデータパックの "money" と確実に衝突する
❌ score, temp, data # 一般名詞は全世界の初心者と衝突する
```

区切り文字は `.` 派（Smithed/Bookshelf）と `_` 派（GM4）があるが、どちらでもよいので**統一**する。
`.` はセレクタ内でも合法で、ドメイン風に階層化できるので個人的には `.` を推奨:

```
nw.casino.balance
nw.casino.bet
nw.pvp.kills
nw.pvp.deaths
```

## 1.5 フェイクプレイヤー（スコアホルダー）の命名

コミュニティで最も確立している規約のひとつ。GM4標準・Bookshelf規約がほぼ同じルールを採用している:

| プレフィックス | 意味 | 例 |
|---|---|---|
| `#` | **非公開（内部用）**。チャットのスコア表示に出ない | `#tmp`, `#nw.atm.result` |
| `$` | **公開API/外部から読み書きしてよい変数** | `$nw.config.max_bet` |
| `#<数値>` | **定数**。値と同じ名前を付ける | `scoreboard players set #100 nw.const 100` |

- `#` 始まりのフェイクプレイヤーはサイドバー等に表示されないため、内部変数は必ず `#` を付ける。
- 定数は `#2`, `#100`, `#-1` のように**値そのものを名前にする**（GM4標準）。`#two` などと訳さない。
- 定数以外の変数はsnake_caseで意味のある名前を付ける: `$has_soul_fire_heatsource`（GM4の実例）。
- フェイクプレイヤー名にも名前空間を付ける（`#nw.foo`）。ライブラリ系データパックとobjectiveを
  共有する可能性がゼロでも、癖として付けておくと事故らない。

## 1.6 エンティティタグ（/tag）の命名

タグは文字数制限がないので、**フル名前空間 + ドット区切り + 分類**が読みやすい。

```
✅ niwatori.casino.dealer          # <名前空間>.<機能>.<役割>
✅ niwatori.entity.marker.atm      # 分類を挟む流儀（entity/block/player等）
✅ bs.persistent                   # Bookshelfの実例: 消してはいけない永続エンティティ
❌ dealer                          # 衝突不可避
❌ DEALER, Dealer                  # 大文字は合法だが、全小文字統一が主流
```

「グローバルに意味を持つ共通タグ」も存在する（10章参照）: `smithed.entity`（データパック製エンティティ）、
`smithed.strict`（他パックは触るな）、`global.ignore`系など。自作markerには必ず自分のタグを付け、
`kill @e[type=marker]` のような雑な掃除で他パックを壊さない/壊されないようにする。

## 1.7 storageの命名

storageはIDそのものが名前空間付き（`namespace:path`）なので追加プレフィックス不要。
ただし**用途別にstorageを分ける**規約を作ると読みやすい:

```
niwatori:in      # 関数呼び出しの引数入れ（マクロ用）※6章
niwatori:out     # 関数の戻り値入れ
niwatori:data    # 永続データ（プレイヤーDB等）
niwatori:temp    # 一時領域（tick内で使い捨て）
niwatori:const   # 定数テーブル（loadで初期化）
```

rx-modules/PlayerDBのような著名パックも `rx.playerdb` 式の名前空間で同様の分離をしている。
キー第一階層はモジュール名にする（Bookshelf: `bs:data <module>.<feature>`）。

## 1.8 カスタムNBT（custom_data）の命名

アイテムやエンティティに独自データを持たせるときは、**必ず自分の名前空間キーで1段ラップ**する。

```
✅ give @s diamond_sword[custom_data={niwatori:{item_id:"rating_sword",tier:3}}]
❌ give @s diamond_sword[custom_data={item_id:"rating_sword",tier:3}]
```

トップレベルに `item_id` のような一般名を置くと、他のデータパックの `custom_data` 判定と誤爆する。
これはMC Datapacks規約の名前空間規約（「NBTも名前空間化せよ」）そのもの。

## 1.9 翻訳キー・チーム・bossbar等その他ID

- bossbar ID: `niwatori:casino_jackpot`（もともと名前空間付き）
- チーム名: 16文字制限あり。`nw.red` のように短縮プレフィックス。
- 進捗/loot table/predicate等のJSON系はすべてリソースパスなので1.1〜1.2と同じ規則。

## 1.10 命名の一般原則（リーダブルコードとの共通部分）

- **具体的な単語を選ぶ**: `check` より `validate_bet`、`do` より `apply_damage`。
- **スコープが広いものほど長い名前**: tick毎に走る主要関数やAPI関数は説明的に。
  数行のヘルパーで同フォルダからしか呼ばれないものは短くてよい。
- **単位・型を名前に含める**: `cooldown_ticks`, `price_tama`（通貨単位タマなら）, `angle_deg`。
  scoreboardはintしか持てないので「固定小数点で100倍した値」なら `speed_x100` 等と明示する。
- **否定形を避ける**: `is_not_ready` より `is_ready` を定義して `unless predicate` で使う。
- **予約語的な名前を知っておく**: Bookshelfは `__load__` `__unload__` `__help__` を
  特殊関数として予約している。自分のパックでも「全モジュール共通の決まったファイル名」を
  少数決めておくと迷子にならない（11章）。

# 2. 名前空間の設計

## 2.1 名前空間はブランドである

名前空間は「フォルダ名」ではなく「あなたの署名」。Smithed規約は
**ファイルパス・タグ・スコア・チーム・storageすべてに一貫したプレフィックス**を要求している。
パック名/作者名/略称のいずれか、一貫して自分に紐づくものを選ぶ。

```
tcc          # "The Creeper's Code" の略（Smithedの実例）
rx.playerdb  # 作者グループ rx-modules + パック名（実例）
gm4_xxx      # Gamemode 4 のモジュール群
bs           # Bookshelf
```

推奨パターン:

- 単発パック: `niwatori_casino` のようなパック名そのまま
- 同一作者の複数パック群: `nw.casino` / `nw.pvp` のように **作者略称.モジュール** で階層化
- scoreboard等の16文字制限があるものは2〜4文字の短縮形（`nw`）、
  ファイルパスやタグはフル（`niwatori`）と、**長短2つの正式名を最初に決めて表に書いておく**

## 2.2 minecraft名前空間は「上書き専用」

`data/minecraft/` に置いてよいのは:

- `tags/function/load.json`, `tags/function/tick.json` への **追記**
- バニラのloot table / recipe / tag等を意図的に置き換える場合

やってはいけないこと（MC Datapacks規約が明示的に禁止）:

- `tick.json` / `load.json` に `"replace": true` を設定する
  → 共存する他のデータパック全てのtickを殺す。サーバー運営では致命傷。
- 自作関数を `data/minecraft/function/` に置く（衝突・上書き事故のもと）

```json
// ✅ data/minecraft/tags/function/tick.json
{
  "values": [
    "niwatori:core/tick"
  ]
}
```

## 2.3 1データパック=1名前空間が基本、例外は「意図的な公開I/F」

基本は自分の名前空間1つ。例外として、他パックとの連携用グローバル名前空間
（`global:` や `#smithed.xxx` のような規約タグ）に参加する場合のみ複数名前空間を持つ。
「なんとなく機能ごとに名前空間を分ける」のは、内部向けなら**フォルダ分けで十分**。
名前空間の数はインターフェースの数、フォルダはモジュールの数、と考える。

# 3. ディレクトリ構造と関数の分割

## 3.1 レイヤーを分ける: エントリポイント / API / 内部実装

読める大規模データパックに共通するのは「**どこから実行が始まり、どこが外部公開で、
どこが触ってはいけない内部か**」が構造で分かること。IMP-Docのアクセス修飾子
（`@public` / `@api` / `@within` / `@private`）の考え方をフォルダ構造に落とすとこうなる:

```
data/niwatori/function/
├── load.mcfunction            # エントリポイント: #minecraft:load から
├── tick.mcfunction            # エントリポイント: #minecraft:tick から
├── api/                       # 他パック・コマブロから呼んでよい公開API
│   ├── give_money.mcfunction
│   └── open_atm.mcfunction
├── casino/                    # モジュール（機能単位）
│   ├── atm/
│   │   ├── open.mcfunction
│   │   ├── deposit.mcfunction
│   │   └── _render_balance.mcfunction    # 先頭 _ = このフォルダ内部専用
│   └── slot/
├── pvp/
│   ├── rating/
│   └── arena/
├── core/                      # 全モジュール共通の基盤（tick分配、時計、定数）
│   ├── tick.mcfunction
│   └── constants.mcfunction
└── debug/                     # デバッグ用（配布時に消せる独立性を保つ）
```

ポイント:

- **tick.jsonに直接ぶら下げる関数は1つだけ**にして（datapack.wikiも推奨）、
  その中で各モジュールのtickへ分配する。tickの全体像が1ファイルで読める。
- 内部専用関数は `_` プレフィックスや `impl/` `zz_` フォルダで「呼ぶな」を表現する
  流儀がある（mcfunctionに言語レベルのprivateはないため、命名で代用する）。
- `debug/` `test/` は本体から参照されない形で分離し、リリース時に丸ごと消せるようにする。

## 3.2 1ファイル=1責務、「executeの都合」による分割を恐れない

mcfunctionでは以下の理由で**ファイル分割が他言語より頻繁に必要**になる:

1. `execute as @e[...] run function ...` で実行者を切り替える（@s化）
2. 条件分岐のブロック化（if文の中身に相当する部分を別ファイルへ）
3. マクロ行を含む部分の分離（6章）
4. 再帰ループの本体

これは「悪」ではなく、mcfunctionにおける関数抽出（Extract Method）そのもの。
むしろ**「このファイルは何の文脈で実行されるか」が一意になるように割る**のが正しい。
1ファイル内に「サーバーとして実行される行」と「各プレイヤーとして実行される行」が
混ざっている方がよほど読めない。

```
❌ tick.mcfunction 内で
execute as @a[predicate=nw:is_in_casino] at @s run tp ...
execute as @a[predicate=nw:is_in_casino] at @s run particle ...
execute as @a[predicate=nw:is_in_casino] at @s run playsound ...

✅ tick.mcfunction:
execute as @a[predicate=nw:is_in_casino] at @s run function nw:casino/player_tick
   casino/player_tick.mcfunction:  (@s = カジノ内プレイヤー, at @s)
tp ...
particle ...
playsound ...
```

セレクタ評価が1回になるので**速度も上がる**（9章）。可読性と性能が同じ方向を向く珍しい言語。

## 3.3 フォルダ名は複数形/カテゴリ名、ファイル名は単数の動作

```
casino/games/slot/spin.mcfunction
casino/games/poker/deal.mcfunction
```

vanillaデータ（loot_table等）の構造も参考になる。`chests/`, `entities/` のように
バニラはカテゴリフォルダを複数形にしている。

## 3.4 function以外のリソースも同じ木構造にそろえる

predicate / advancement / loot_table を、対応するfunctionと**同じサブパス**に置くと
関連が追いやすい:

```
data/niwatori/function/casino/atm/open.mcfunction
data/niwatori/predicate/casino/is_in_casino.json
data/niwatori/advancement/casino/atm/interacted.json
data/niwatori/loot_table/casino/reward/jackpot.json
```

進捗トリガー→`rewards.function`→関数、の連鎖はデータパックの主要イベント機構なので、
`advancement/casino/atm/interacted.json` と `function/casino/atm/on_interact.mcfunction` のように
**対になる名前**を付ける（イベントハンドラ命名: `on_<イベント>`）。

## 3.5 生成ファイルは手書きファイルと分ける

Pythonスクリプト等でloot tableやmcfunctionを自動生成する場合（チェスト戦利品テーブル等）、
生成物は `generated/` サブフォルダに隔離し、フォルダ先頭に
`# THIS FILE IS GENERATED BY tools/chest-loottable.py -- DO NOT EDIT` を入れる。
Bookshelfもビルド時生成ファイルを明確に区別している。生成物を手で直すと次の生成で消えるので、
「手で編集してよい領域」と「スクリプトの出力先」を物理的に分離するのが鉄則。

# 4. コメントとドキュメンテーション

## 4.1 全関数にヘッダコメントを書く（GM4標準）

Gamemode 4は**テンプレート由来以外の全関数**に以下のヘッダを義務付けている。
mcfunctionには引数リストも型もdocstringも無いので、これが唯一の「シグネチャ」になる:

```mcfunction
# Zauber大釜の生成時にパーティクル演出を再生する
# @s = 付呪本を持っているプレイヤー
# at @s (align xyz)
# run from niwatori:cauldron/create
```

最低限書くべき4点:

1. **何をする関数か**（1行説明）
2. **@s が誰か**（セレクタ表記 or 日本語。複雑なセレクタは日本語で）
3. **実行位置/回転**（`at @s`, `positioned 0 64 0`, alignの有無など）
4. **どこから呼ばれるか**（`run from nw:xxx`、複数なら「functions in nw:till/blocks/」、
   scheduleなら「scheduled by nw:xxx」）

「どこから呼ばれるか」はgrepでも調べられるが、ヘッダにあることで
**開いた瞬間に文脈が復元できる**。これがmcfunction可読性の生命線。

## 4.2 IMP-Doc: 機械可読なドキュメント規約

日本コミュニティ発で広く使われるドキュメント仕様 **IMP-Doc**（Interoperable Module Protocol
Documentation）は、上のヘッダコメントを形式化したもの。先頭行を `#>` で始める:

```mcfunction
#> niwatori:casino/atm/deposit
# ATMへの入金処理。所持金スコアをstorageの口座残高へ移す。
#
# @within function niwatori:casino/atm/menu_handler
# @input
#   score @s nw.deposit_amount : 入金額（タマ）
# @output
#   storage niwatori:data account[].balance : 更新後残高
# @private
```

主なアノテーション（IMP-Doc仕様 + DHP拡張）:

| アノテーション | 意味 |
|---|---|
| `@input` / `@output` | 実行前に設定すべき値 / 実行後に設定される値 |
| `@within <パス>` | 指定した場所からしか呼ばれない（実質private） |
| `@private` / `@public` / `@api` | アクセス修飾子 |
| `@user` | ユーザー（プレイヤーやコマブロ）が直接実行してよい |
| `@context` | 要求する実行文脈（位置・回転・実行者） |
| `@handles` / `@patch` | イベント処理 / 他リソースへのパッチ |
| `@deprecated` / `@internal` | 非推奨 / 内部用（DHP拡張） |

エディタ（Spyglass / Datapack Helper Plus）がこれを解釈して補完・参照検索に使える。
公式仕様ではなくただのコメントなので、ゲーム動作には一切影響しない=導入コストゼロ。

## 4.3 処理ブロックごとの節コメント

Elliversのプラクティス集が示す標準形。空行+コメントで段落を作る:

```mcfunction
#> niwatori:pvp/arena/start
# 試合開始処理
# @within function niwatori:pvp/lobby/countdown

# 参加者を初期化
scoreboard players set @a[tag=nw.pvp.player] nw.pvp.kills 0
effect clear @a[tag=nw.pvp.player]

# アリーナへ転送
execute as @a[tag=nw.pvp.player] run function niwatori:pvp/arena/teleport_to_spawn

# 開始演出
title @a[tag=nw.pvp.player] title {"text":"FIGHT","color":"red"}
```

## 4.4 コメントの内容: 「何を」より「なぜ」

コマンドの直訳コメント（`# プレイヤーにダイヤを与える` → `give @s diamond`）は書かない。
書くべきは**コマンドから読み取れない理由・制約・ハック**:

```mcfunction
# NG: tpでY+0.5している           ← 見れば分かる
# OK: マーカーが半ブロック床にめり込むとレイキャストが失敗するためY+0.5補正
tp @s ~ ~0.5 ~
```

特にmcfunctionは「1tick遅延しないと反映されないNBT」「セレクタの評価順」「マクロの再コンパイル」
などの罠ハック密度が高いので、**未来の自分への言い訳**を必ず残す。

## 4.5 #region による折りたたみ

長い分岐列挙は `#region` / `#endregion`（DHP/一部エディタが折りたたみ対応）でグループ化できる:

```mcfunction
#region スロット絵柄ごとの配当分岐
execute if score #reel nw.tmp matches 1 run function nw:casino/slot/payout/cherry
execute if score #reel nw.tmp matches 2 run function nw:casino/slot/payout/bell
execute if score #reel nw.tmp matches 3 run function nw:casino/slot/payout/seven
#endregion
```

ただし折りたたみが必要なほど長い列挙は、そもそも分岐テーブル関数への分割やマクロ化を検討するサイン。

## 4.6 コメントも保守対象

Bookshelfのガイドラインが強調する点: **挙動を変えたらコメントも更新する**。
古いコメントは無いより悪い。特に `@input`/`@output` とヘッダの「run from」は
リファクタ時に嘘になりやすいので、レビュー/セルフチェック項目に入れる（16章）。

# 5. 実行文脈（execution context）の設計

## 5.1 「@s中心主義」で書く

mcfunctionの可読性を最も左右するのは実行文脈の管理。原則:

1. **関数の中では@sと相対座標だけを使う**（誰に対して動くかは呼び出し側が決める）
2. セレクタで対象を選ぶのは**呼び出し側（分配層）の責務**
3. 文脈を変えるexecuteが3行以上続いたら関数に切り出す

これにより関数が「対象非依存の部品」になり、テストも再利用も楽になる。
Bookshelfの「Conservation Principle」も同じ思想で、**関数は入力に責任を持ち、
出力以外を書き換えない**（勝手にスコアを消したり作ったりしない）ことを求めている。

## 5.2 文脈をヘッダに書く（4章と連動）

`# @s = ...` `# at @s` が書いてあれば、`~ ~ ~` が誰の足元かが一目で分かる。
文脈が「サーバー（実行者なし）」の関数では、うっかり`@s`や相対座標を使わないこと。
loadやscheduleから呼ばれる関数は実行者・位置を失う（ワールドスポーン基準になる）ことを
ヘッダに明記しておくと事故が減る:

```mcfunction
#> niwatori:core/minute_clock
# scheduleで毎分self-rescheduleされる時計
# ⚠ 実行文脈なし: @sは存在せず、位置はワールドスポーン。エンティティ操作禁止
```

## 5.3 anchoredとalignは使った場所でコメント

`anchored eyes` や `align xyz` は後続コマンドの座標の意味を静かに変える。
使う場合は同一行 or 直前コメントで意図を書く。

# 6. マクロ関数の設計 ★

1.20.2で追加されたマクロ（`$(name)` 展開）は強力だが、
「**マクロ行を含む関数は呼び出し時に毎回再解析される**」という性質があり、
datapack.wikiやコミュニティの共通見解として**通常のコマンドより大幅に遅い**。
さらに「マクロを使う行は必ず専用ファイルに分けたくなる」ため、設計規約が無いと
ファイルが散らかる。ここが本ガイドの目玉。

## 6.1 命名規約: 「本体」と「マクロ実体」をペアにする

マクロのために分離したファイルは、**呼び出し元と同名+固定サフィックス/固定サブフォルダ**にする。
おすすめは2案。プロジェクトでどちらかに統一する:

**案A: 同名フォルダ + macro.mcfunction（推奨）**

```
casino/atm/deposit.mcfunction          # 引数を準備して呼ぶ「本体」
casino/atm/deposit/macro.mcfunction    # $(...)を含む「マクロ実体」
```

**案B: サフィックス方式**

```
casino/atm/deposit.mcfunction
casino/atm/deposit_macro.mcfunction    # または deposit.m.mcfunction
```

どちらでも「`deposit` を読みたい人が迷わず両方を見つけられる」ことが目的。
逆に **マクロ実体を `util/` や `macro/` などの遠いフォルダにまとめて置かない**こと。
処理のまとまり（モジュール）が壊れ、コードジャンプが往復になる。

さらにIMP-Docの `@within` を書けば「この`macro`は隣の`deposit`からしか呼ばれない」ことが
機械にも人間にも伝わる:

```mcfunction
#> niwatori:casino/atm/deposit/macro
# 実際の入金額を反映するマクロ実体
# @within function niwatori:casino/atm/deposit
# @input
#   args: {amount: int, unit: string}
$scoreboard players remove @s nw.money $(amount)
$tellraw @s {"text":"$(amount) $(unit) を入金しました"}
```

## 6.2 引数受け渡しの呼び出し規約を1つに固定する

マクロの引数は「インラインNBT」「with storage」「with entity」「with block」の4通りあるが、
プロジェクト内の規約としては次の2択に絞ると読みやすい:

1. **定数引数** → インラインNBTで直書き（呼び出し箇所を見れば値が分かる）
   ```mcfunction
   function nw:casino/atm/deposit/macro {amount: 100, unit: "タマ"}
   ```
2. **動的引数** → 専用storage `nw:in` の**関数パスと同じキー**に詰めて `with` で渡す
   ```mcfunction
   # 呼び出し側（本体 deposit.mcfunction）
   execute store result storage nw:in casino.atm.deposit.amount int 1 \
       run scoreboard players get @s nw.deposit_request
   data modify storage nw:in casino.atm.deposit.unit set value "タマ"
   function nw:casino/atm/deposit/macro with storage nw:in casino.atm.deposit
   ```

「引数はどこから来たのか」問題（マクロ最大の可読性リスク)を、
**`nw:in` の `casino.atm.deposit` を見れば必ず分かる**という形で解決する。
Bookshelf等のライブラリも「入力→専用の場所に置く→呼ぶ」の規約で統一している。
戻り値が要る場合は同様に `nw:out` へ書く or `return run` で数値を返す。

## 6.3 マクロ行は最小限に隔離する

再解析コストは**マクロ行を含むファイル全体**にかかる。したがって:

```
❌ 40行の処理の中に $行が2行混ざったファイル
✅ 38行の通常関数 + $行2行だけのマクロ実体ファイル
```

「$が1行でもあるならそのファイルは短く」を徹底する。
また、**マクロで実現できても、分岐数が少なく固定なら普通のexecute分岐の方が速い**。

```
❌ 3種類しかない絵柄をマクロで動的ディスパッチ
✅ execute if score ... matches 1 run function nw:...cherry （3行書く）
```

マクロが正当化されるのは「値が実行時にしか決まらない」「組み合わせが多すぎて列挙不能」
「tellraw/give等の文字列に動的値を埋める」場合。
tick毎に走るホットパスからのマクロ呼び出しは設計から見直す。

## 6.4 マクロ引数名の命名

- `$(a)` `$(x)` のような1文字は禁止。呼び出し側のNBTキーがそのままドキュメントになるので
  `$(amount)` `$(target_uuid)` `$(dimension)` のように**キー名=引数名**として意味を持たせる。
- 単位を含める: `$(delay_ticks)`, `$(price_tama)`。
- 同じマクロ実体を複数箇所から呼ぶ場合、引数構造（キー名と型）はヘッダの `@input` に必ず書く。
  マクロは引数不足だと**実行時に初めて失敗する**（コンパイル時チェックがない）ため、
  ドキュメントが型定義の代わりになる。

## 6.5 セキュリティ: ユーザー入力をマクロに流さない

`$$(code)` のような書き方や、プレイヤーが編集可能なNBT（記名アイテム、本の内容、
custom_data）を無検証でマクロに渡すと、**任意コマンド実行（ACE）**が成立し得る。
コミュニティでも実証されている既知の穴。サーバー運営では特に:

- マクロに渡す文字列は自分のデータパックが書き込んだものだけに限定する
- プレイヤー由来の文字列を使うなら、事前にホワイトリスト照合（スコアIDやenum化）を挟む
- 「文字列そのもの」ではなく「ID番号→loadで初期化したテーブル参照」で間接化する

## 6.6 マクロvs代替手段の選択表

| やりたいこと | 第一候補 | マクロを使うべき場合 |
|---|---|---|
| 少数の固定分岐 | execute if + 分岐関数 | 分岐が数十以上で列挙が破綻する時 |
| 動的な数値をコマンドへ | execute store / scoreboard | tp座標やeffect時間など store不可の引数 |
| 動的テキスト表示 | tellrawのscore/nbt/selector埋め込み | 文体自体が変わる・翻訳分岐など |
| アイテム動的生成 | loot table + item_modifier | NBT構造が実行時にしか決まらない時 |
| エンティティ指定 | tag付け→セレクタ | UUIDを直接コマンドに埋める必要がある時 |

「マクロでしかできないこと」以外にマクロを使わない、が結論。

# 7. 変数管理（scoreboard / storage / マクロ引数）

## 7.1 使い分けの原則

| 手段 | 型 | 速度 | 用途 |
|---|---|---|---|
| scoreboard | int32のみ | 最速 | カウンタ、フラグ、演算、tick毎の状態 |
| storage | 任意NBT | 速い（entity/block NBTより大幅に） | 構造化データ、文字列、リスト、設定 |
| entity/block NBT | 任意 | **遅い**（特にプレイヤー） | 最終読み書きのみ。作業領域にしない |
| マクロ引数 | 任意 | 呼び出しが遅い | 関数への引数渡し専用 |

NBTを何度も加工するときは「一度storageへコピー→storage上で加工→書き戻し」が定石
（Elliversのプラクティス集）。

## 7.2 objectiveは増やしすぎない: フェイクプレイヤー活用

「変数1個ごとにobjectiveを1個作る」のは典型的アンチパターン。
グローバル変数はobjective 1〜2個（`nw.var` 等）+フェイクプレイヤー名で無限に作れる:

```mcfunction
scoreboard players set #casino.jackpot nw.var 0
scoreboard players set #arena.state nw.var 1
```

objectiveを分けるべきなのは「エンティティごとに値を持つ」「trigger用」「statistics連動」の時。

## 7.3 定数はloadで初期化する定数プール

乗除算に使う定数は `nw.const` objectiveに `#値` 名で集中管理（1.5参照）。
loadで必ず初期化し、**実行中にsetし直さない**:

```mcfunction
#> niwatori:core/constants
# @within function niwatori:load
scoreboard players set #2 nw.const 2
scoreboard players set #100 nw.const 100
scoreboard players set #20 nw.const 20    # 1秒 = 20tick
```

## 7.4 「0と未定義は違う」を設計に織り込む

`matches ..0` は未定義スコアにマッチしない。テストワールドでは動くのに本番で動かない
バグの代表格。**プレイヤー系スコアは参加時/loadで明示的に0初期化**する規約にする:

```mcfunction
# 新規参加者の初期化（tickで未初期化者だけ拾う）
execute as @a[predicate=!nw:is_initialized] run function nw:core/init_player
```

## 7.5 一時変数の規約

- 名前: `#tmp`, `#tmp2` ではなく `#calc.damage`, `#calc.rate` のように用途を書く。
  ただし**関数内で完結し、次のtickに持ち越さない**ものだけを一時変数と呼ぶ。
- 使い終わった一時storageは放置でよい（次回上書きされる設計にする）。
  逆に `data remove` で几帳面に消して回るとコスト増+他所が読んでいた場合に事故る。
  Bookshelfの保存原則: **スコアの削除は極力しない、リセット/上書きで済ます**。

## 7.6 enumは数値+コメント表 or 「Enum Scoreboard」

状態機械（ゲーム進行state等）は文字列でなくintで持ち、対応表をコメントとloadに残す:

```mcfunction
#> niwatori:pvp/arena/states
# #arena.state nw.var の値:
#   0 = 待機 / 1 = カウントダウン / 2 = 試合中 / 3 = リザルト
```

MC Datapacksコミュニティには、値名をフェイクプレイヤー化してobjective自体を
enum定義として使う「Enum Scoreboard」というテクニックもある
（`scoreboard players set WAITING nw.arena_state_enum 0` のように定義を実体化する）。

# 8. 制御構造のイディオム

## 8.1 早期リターン（return）を使う

1.20.2+の `/return` はガード節そのもの。ネストしたexecuteより読める:

```mcfunction
#> niwatori:casino/atm/withdraw
# 出金。残高不足なら即終了
execute if score @s nw.money < #withdraw.amount nw.var run return run \
    tellraw @s {"text":"残高が足りません","color":"red"}

# ここから正常系
...
```

Minecraft Wiki最適化チュートリアルも「後続条件があり得なくなったら`return run`で打ち切る、
順序に意味がなければ**成立確率の高い条件を先に**書く」ことを推奨。
注意: `execute as @e ... run return` は最初の1体で関数全体が終わるので、
複数対象への分岐がある関数では使わない（分岐専用関数に隔離してから使う）。

## 8.2 if/else は「フラグ+2連if」か「return」で

mcfunctionにelseは無い。素朴な2連 `if`/`unless` は1行目が状態を変えると2行目も走る事故がある。
定石は (a) 結果を一時スコアに保存して分岐、(b) return方式:

```mcfunction
# (a) 判定を先に固定する
execute store success score #can_pay nw.tmp if score @s nw.money matches 100..
execute if score #can_pay nw.tmp matches 1 run function nw:casino/slot/spin
execute if score #can_pay nw.tmp matches 0 run tellraw @s {"text":"コインが足りない"}
```

## 8.3 ループは3種類を使い分け、必ず停止条件を先頭に

| 種類 | 実装 | 用途 |
|---|---|---|
| 毎tickループ | `#minecraft:tick` | 常時監視。最小限に |
| 低頻度ループ | `schedule function ... 20t` の自己再スケジュール | 秒単位で十分な監視・時計 |
| tick内再帰 | 関数が自分を`function`で呼ぶ | カウント処理、レイキャスト |

再帰は**必ず1行目に脱出条件**（ガード）を書く。条件なし再帰はmaxCommandChainLength
まで暴走する。レイキャストの1ステップ距離は貫通防止のため0〜1ブロック程度にする
（コミュニティの定番注意点）。scheduleは実行文脈を失う点もヘッダに明記（5.2）。

## 8.4 分岐テーブルは「ディスパッチ関数」に隔離

同じ対象への `execute if score ... run function ...` の列挙は、それ専用の
`dispatch.mcfunction` / `payout/_dispatch.mcfunction` を作ってそこに置く。
tick関数や業務ロジックの中に分岐列挙を埋め込まない。

## 8.5 イベント駆動を最優先する（ポーリングは最後の手段）

「毎tick全員を調べる」前に、advancementトリガー（`player_interacted_with_entity`,
`placed_block`, `inventory_changed`, `using_item`...）+ `rewards.function` で
**起きた時だけ関数を走らせる**設計を検討する（Elliversも同旨）。
進捗は `revoke` して使い捨てるイディオムとセットで:

```mcfunction
#> niwatori:casino/atm/on_interact
# advancement niwatori:casino/atm/interacted の rewards から
# @s = ATMエンティティをクリックしたプレイヤー
advancement revoke @s only niwatori:casino/atm/interacted
function niwatori:casino/atm/open
```

triggerスコアボード（dialogの`run_command`権限回避等）も同様にイベント駆動の一種。
`tick監視 → scoreboard/predicate → advancement/trigger` の順で「より受動的な仕組み」へ
置き換えられないか常に考える。

# 9. パフォーマンス作法

読みやすさと速度が両立するのがデータパックの良いところ。Minecraft Wikiの
「Tutorial: Optimizing a data pack」とコミュニティ実践からの要点。

## 9.1 セレクタ

- `@e` は最小限。**型を必ず指定**（`@e[type=marker]`）し、可能なら `tag=` で第一絞り込み。
  セレクタ引数は概ね「安い順に絞れる」よう `type` → `tag` → `distance`/`scores` の意識で。
- 同じセレクタを複数行で繰り返さない → `as`で関数に入って`@s`化（3.2）。
- 遠くで動く必要のない処理には `distance=..N` を付ける。大量エンティティ相手なら
  「プレイヤー周辺に定期的に`active`タグを付け、以後は`tag=active`で選ぶ」方式が
  distance連打より安い（Wiki推奨テク）。
- `@a` は `@e[type=player]` より速い（プレイヤーリストだけ走査する）。

## 9.2 NBTを避ける階段

1. セレクタ`nbt=`引数 → **predicateへ**（equipment, vehicle等は専用フィールドがある）
2. アイテム判定 → `execute if items entity @s weapon.mainhand <item>`（1.20.5+、最速級）
3. どうしてもNBT加工 → storageへコピーして加工（プレイヤーNBT直接操作は最悪コスト）
4. アイテム改変 → `data modify`でなく **item_modifier / item replace**

なお「predicateの中で`nbt`キーだけ使う」のは速くならない（Ellivers）。
predicateが速いのは構造化フィールドを使った時だけ。

## 9.3 tickを軽くする

- tick.jsonには分配関数1つ（3.1）。分配関数の先頭で「そもそも今やることがあるか」を
  スコア1個で判定して早期return。
- 毎tick必要ない処理は `schedule` 自己ループで 2t/5t/20t に間引く。
  エンティティ個別なら predicate の `periodic_tick` も使える（セレクタは走るのでscheduleより劣る）。
- 重い計算は複数tickに分割する（時分割）。tick予算(50ms)を意識する。

## 9.4 計測してから最適化する

- `/debug start` → `/debug stop` のプロファイル結果で関数ごとの時間を見る
  （`commandFunctions`/`scheduledFunctions` の項目）。
- F3のTPSグラフ/`/perf`、または計測用データパック（Kragast's Benchmark等）でベンチ。
- 推測で「速い書き方」に書き換えて可読性を落とすのは本末転倒。**遅いと分かった場所だけ**直す。
  この点も『リーダブルコード』と同じ。

# 10. 互換性・複数データパック共存

サーバー（にわとりMCのようにロビー+PVP等）で複数パックを同居させるなら必須の章。

## 10.1 絶対ルール

- `minecraft`名前空間のtick/loadタグに `"replace": true` を書かない（2.2）
- 全ての名前（objective/tag/storage/team/bossbar/fakeplayer）に名前空間プレフィックス
- バニラ挙動の上書き（loot table等）は「合理的な範囲」に留め、README/descriptionに明記する

## 10.2 共通タグ規約に乗る

- `smithed.entity`: データパックが生成したエンティティ（「バニラのゾンビ」等を対象とする
  他パックの処理から除外してもらう）
- `smithed.strict`: 他パックは一切触らないでほしいエンティティ
- Bookshelfの `bs.persistent` のように「killしないで」系タグを尊重する
  （掃除コマンドは `kill @e[type=item]` でなく `kill @e[type=item,tag=!nw.keep,tag=!smithed.strict]` 的に）

自作エンティティには自分のタグ+`smithed.entity`を両方付けるのが行儀の良い作法。

## 10.3 カスタムアイテムの識別

`custom_data` に自名前空間で `{niwatori:{id:"rating_sword"}}` 形式のIDを持たせ、
判定は必ずこのIDで行う。Item名やCustomModelDataでの判定は、リソパ更新や
他パックとの併用で壊れる。（旧CMD時代はSmithedのCMDプレフィックス規約があったが、
現行は`custom_data`/`item_model`コンポーネントで名前空間分離するのが素直。）

# 11. load / tick / インストール / アンインストール

## 11.1 標準ファイルセットを決める

良いデータパックは「入口」が全部同じ場所にある。推奨テンプレート:

```
function/
├── load.mcfunction        # #minecraft:load から。冪等に書く
├── tick.mcfunction        # #minecraft:tick から。分配のみ
├── install.mcfunction     # 初回セットアップ（forceload, marker設置等）
├── uninstall.mcfunction   # 完全撤去
└── version.mcfunction     # バージョン表示（任意）
```

Bookshelfは `__load__` / `__unload__` / `__help__` をモジュール毎の予約名にしている。
名前は何でもよいが**全モジュールで統一**する。

## 11.2 loadは冪等（何度実行しても安全）に

`/reload` のたびに走るので、loadで「追加」する処理は既存チェック付きにする:

```mcfunction
#> niwatori:load
# ⚠ /reloadごとに実行される。冪等であること
scoreboard objectives add nw.var dummy
scoreboard objectives add nw.money dummy "所持金"
function niwatori:core/constants
schedule function niwatori:core/minute_clock 1t replace   # replaceで多重登録防止
tellraw @a[tag=nw.admin] {"text":"[にわとりMC] reloaded vX.Y.Z","color":"gray"}
```

`schedule ... replace` を使わないとreloadのたびにループが増殖する。定番バグ。
reload完了メッセージ（バージョン付き）は運用上のデバッグ効率を大きく上げる
（MC Datapacksスタイルガイドの「Reloading Message」）。

## 11.3 uninstallを提供する

MC Datapacks規約「Datapack Uninstallation」。行儀の良いパックは
自分が作ったものを全部消す関数を持つ:

```mcfunction
#> niwatori:uninstall
# このデータパックの痕跡を全て除去する。実行後にdatapack disable推奨
scoreboard objectives remove nw.var
scoreboard objectives remove nw.money
kill @e[tag=niwatori.entity]
data remove storage niwatori:data {}
schedule clear niwatori:core/minute_clock
tellraw @s {"text":"[にわとりMC] アンインストール完了。datapack disableしてください"}
```

サーバー運営では「入れて試して消す」が頻繁なので、uninstallの有無が事故率に直結する。

# 12. バージョン管理・配布・メタ情報

## 12.1 pack.mcmeta

- `description` にはパック名・対象MCバージョン・自分のバージョン・作者を入れる
  （データパック選択画面とdatapack listで見える唯一の自己紹介）。
- 1.21.9以降（pack_format 88〜）は `min_format` / `max_format` 方式に移行済み。
  1.21.11なら新旧両方の書式に注意。26.x系（pack_format 100+）への更新時は
  フォーマット変更点（NBT型厳格化等）の洗い出しをリリースノートで確認する。

```json
{
  "pack": {
    "description": "にわとりMC Casino v1.4.0 (MC 1.21.11) by niwatori",
    "min_format": 88,
    "max_format": 88
  }
}
```

## 12.2 バージョン番号をデータでも持つ

`load`時に `#version.major nw.var` 等へ格納しておくと、
ワールドに残った旧データからのマイグレーション（「v1.3のstorage形式をv1.4へ変換」）が書ける。
GM4がモジュール更新リストで同様の仕組みを持つ。**storage構造を変えたら移行関数を書く**、
を運用ルールにする。

## 12.3 Gitで管理する

- リポジトリのルートは「pack.mcmetaがある階層」（エディタ支援ツールの前提でもある）。
- zipではなくフォルダのままworldに置き、シンボリックリンク or デプロイスクリプトで反映。
- コミット規約（Bookshelfは Conventional Commits 風）を軽く決めておくとchangelogが自動化できる。

- **VSCode + Spyglass（旧Datapack Helper Plus）**: 補完・参照検索・リネーム・IMP-Doc解釈。

# 14. テストとデバッグ

## 14.1 debugネームスペース/フォルダを常設する

```
function/debug/
├── dump_player.mcfunction    # @sの主要スコアとstorageをtellraw
├── give_test_kit.mcfunction
└── reset_all.mcfunction
```

- デバッグ出力は `@a[tag=nw.debug]` にだけ流す（本番プレイヤーに漏らさない）。
- Bookshelf Devのように「ログ・変数ダンプ専用の開発用パック」を分離する構成も良い。

## 14.2 表明（アサーション）を書く

「起きないはず」の状態を検知したら管理者へ通報する行を要所に入れる:

```mcfunction
execute if score #arena.state nw.var matches 4.. run \
    tellraw @a[tag=nw.admin] {"text":"[BUG] arena.stateが範囲外","color":"red"}
```

## 14.3 テスト関数

純粋な計算関数（レーティング計算等）は `test/` に期待値比較の関数を書き、
手動 or CI（ヘッドレスサーバー起動）で叩ける形にしておく。
`execute store` + `if score ... matches 期待値` + 結果tellraw、で最低限のユニットテストになる。

# 15. アンチパターン集

| アンチパターン | なぜ悪い | 代わりに |
|---|---|---|
| tick.jsonに関数を直接何個もぶら下げる | 実行順が読めない、全体像が消える | 分配関数1つ（3.1） |
| `replace: true` をtick/load.jsonに | 他パック全滅 | 絶対禁止（2.2） |
| プレフィックス無しobjective/tag | 衝突 | 名前空間プレフィックス（1章） |
| 1関数に複数の実行文脈 | `~ ~ ~`が誰か分からない | @s中心+分割（5章） |
| 巨大マクロ関数 | 全行が毎回再解析 | マクロ行だけ隔離（6.3） |
| ユーザー入力→マクロ直結 | 任意コマンド実行 | ID化・検証（6.5） |
| `nbt={SelectedItem:...}`セレクタ | 激重 | predicate / if items（9.2） |
| 条件なし自己再帰 | チェーン上限まで暴走 | ガード節必須（8.3） |
| scheduleにreplace無し | reload毎にループ増殖 | `replace`指定（11.2） |
| `kill @e[type=marker]`等の雑掃除 | 他パック・自パックの基盤を破壊 | タグで限定（10.2） |
| 0初期化を前提にした`matches ..0` | 未定義スコアに効かない | 明示的初期化（7.4） |
| コメントとコードの乖離 | 嘘ドキュメント | 変更時にヘッダも更新（4.6） |
| デバッグsayの放置 | 本番でチャット汚染 | debugタグ宛てtellraw（14.1） |
| 生成ファイルの手編集 | 次の生成で消える | 生成物隔離（3.5） |

# 16. 提出前チェックリスト

**命名・構造**
- [ ] すべて lower_snake_case か
- [ ] objective / tag / storage / fakeplayer / team に名前空間プレフィックスがあるか
- [ ] tick.jsonに直接ぶら下がる関数は1つか
- [ ] 内部専用関数が `_` / `@within` / フォルダで区別されているか

**ドキュメント**
- [ ] 全関数にヘッダ（説明 / @s / 実行位置 / run from）があるか
- [ ] マクロ実体に `@input`（引数の名前と型）が書かれているか
- [ ] 変更した関数のヘッダコメントは現状と一致しているか

**マクロ**
- [ ] `$` 行を含むファイルは最小か
- [ ] 引数はインラインNBT（定数）か `nw:in`（動的）に統一されているか
- [ ] プレイヤー由来の文字列がマクロに流れ込む経路がないか

**動作・性能**
- [ ] loadは冪等か（objectives add重複、schedule replace）
- [ ] `@e`に type と tag が付いているか / nbtセレクタが残っていないか
- [ ] uninstallで全部消えるか（objective, tag付きエンティティ, storage, schedule, bossbar, team）
- [ ] /debugプロファイルでtick関数の時間を一度は見たか

**互換**
- [ ] minecraft名前空間への追加はtick/load/意図的上書きのみか
- [ ] 自作エンティティにタグ（+smithed.entity）が付いているか
- [ ] pack.mcmetaのformat/description/バージョンが最新か
